01 / Overview

The brief & direction.

This calculator was assigned as my first-week internship project. I chose to go beyond a basic utility interface and build a polished, responsive experience with a glass-inspired visual system.

01

Functional Goal

Support common arithmetic operations, decimals, percentage input, clear, delete, and reliable result handling.

02

Visual Goal

Create a centered premium interface using translucent surfaces, backdrop blur, gradients, soft borders, and depth.

03

Experience Goal

Make the calculator responsive, keyboard-friendly, readable, and pleasant to use across desktop and smaller screens.

02 / Structure

Three clear layers.

I organized the project around the core separation of concerns used in frontend development: HTML for structure, CSS for presentation, and JavaScript for behavior.

HTML

Structure Layer

Defines the wrapper, calculator body, display, keypad, action controls, operators, and semantic page hierarchy.

CSS

Presentation Layer

Controls the responsive grid, glass effect, gradients, background composition, transitions, shadows, and animations.

JAVASCRIPT

Behavior Layer

Handles expression state, button events, keyboard input, validation, deletion, clearing, evaluation, and errors.

03 / Styling

Designing the glass.

Transparency alone does not create convincing glassmorphism. The final appearance comes from layering blur, contrast, borders, lighting, saturation, gradients, and carefully controlled shadows.

Visual System

The calculator body and controls share a consistent material language.

  • Semi-transparent surfaces
  • Backdrop blur and saturation
  • Subtle white borders
  • Cyan and purple accent gradients
  • Layered ambient shadows
  • Soft rounded geometry

Interaction System

Small feedback details make the interface feel responsive and intentional.

  • Lift effect on hover
  • Pressed state on click
  • Smooth CSS transitions
  • Distinct operator styling
  • Prominent equals action
  • Subtle floating animation
04 / Logic

From input to result.

The JavaScript logic keeps track of the current expression, validates important input cases, updates the display, and safely handles invalid calculations without breaking the interface.

01User enters a value
02Event listener detects input
03Input is validated
04Expression state updates
05Result or error is displayed

Decimal Validation

The current number segment is checked before another decimal point is accepted, preventing values such as 12.5.7.

Operator Handling

If consecutive operators are entered, the latest operator can replace the previous one to reduce malformed expressions.

Keyboard Support

Numbers and operators work from the keyboard, while Enter calculates, Backspace deletes, and Escape clears the state.

05 / Challenges

Problems I solved.

The project became more valuable when visual and logic issues appeared. Each challenge pushed me to improve the implementation instead of only completing the minimum requirement.

Visual

The glass looked like plain transparency

Using a transparent background alone did not create enough depth or material realism.

Solution: I combined backdrop blur, saturation, borders, gradients, highlights, and shadows.

Layout

The final keypad row needed correction

The equals button initially occupied more grid space than intended and affected the alignment.

Solution: I corrected the grid span so 0, decimal, and equals remain on the same row.

Logic

Multiple decimal points created invalid numbers

Without validation, a user could add more than one decimal point inside the same number segment.

Solution: I inspect the active numeric segment before accepting another decimal.

Reliability

Invalid expressions could interrupt the experience

Malformed input needed to fail gracefully rather than leaving the calculator in a broken state.

Solution: I added controlled error handling and reset behavior for safe recovery.

Precision

Floating-point results could become unnecessarily long

JavaScript decimal arithmetic can sometimes expose awkward floating-point precision artifacts.

Solution: I normalized displayed precision before presenting the final value.
